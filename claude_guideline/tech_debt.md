# 기술 부채 (Tech Debt) 방지

근본 원인 해결 (정공법) 을 우선하고, 우회·workaround·임시 hack 의 허용 조건과 정리 시점을 정의하는 SSOT.

## 정공법 우선

- 시간이 더 걸리더라도 **근본 원인 해결** 을 선택한다.
- workaround / monkey-patch / 우회 코드 / 임시 hack 은 **기본 금지**.
- 본 원칙은 [coding.md](coding.md) §작업 범위 의 "회피 대안 제시 금지" 보다 강하다 — 우회는 회피의 일종.

## 실시간 / 임베디드에서 정공법이 특히 중요한 이유

- 우회 코드가 실시간 제약 (interrupt latency, jitter) 을 깨뜨릴 수 있음.
- 펌웨어 OTA (Over-The-Air) 가 어려운 환경에서는 정정 비용이 기하급수.
- silicon datasheet 대비 비약 추론이 누적되면 인증 (AEC-Q100 등) 미통과 위험.

## 우회 허용 3 조건

다음 셋을 **모두** 만족할 때만 우회 / workaround 허용. 그 외에는 사용자 명시 승인 후 진행:

1. **시한 강제** — 외부 데드라인이 정공법 시간을 허용하지 않는다.
2. **격리 가능** — 우회 코드가 한 파일·한 함수 안에 격리되며 외부에 영향 없다.
3. **TODO + 사유 기록** — 코드 내 `TODO(YYYY-MM-DD): <사유>` 및 commit message 에 명시.

## 정공법이 사용자 지시 범위를 초과할 때

사용자가 "이 한 줄만 고쳐줘" 라고 했는데 근본 원인이 다른 모듈의 구조 문제인 경우, 정공법을 **자율 실행하지 않는다** (사용자 지시 범위 외 → CLAUDE.md 핵심 원칙 #2 위반).

1. 표면 수정의 효과·리스크를 1줄로 보고.
2. 정공법 비용 / 리스크 / 대안을 2~3줄로 함께 보고 (1줄 대안 한도 — [coding.md](coding.md) §작업 범위 허용 범위 내).
3. 사용자 선택 수신.
4. 정공법 승인 시에만 범위 확장. 미승인이면 표면 수정만.

## 시간 트레이드오프 보고

- 정공법으로 추가 시간이 예상되면 **착수 전** 사용자에게 보고.
- "빠른 수정" 을 사용자가 명시 요청하지 않은 이상 정공법 기본.

## 임시 / 진단 코드 정리

- **기본**: 동일 작업 단위에서 제거 — 디버그 print, 임시 로그, 진단 stub.
- **예외**: 사용자가 명시적으로 "TODO 로 남겨라" / "다음 작업에서 정리" 라고 요청한 경우에 한해 `TODO(YYYY-MM-DD): ...` 형식으로 유지하고 다음 commit message 에 보고.
- 자동 미루기 금지 — 사용자 명시 지시만 수행 (CLAUDE.md 핵심 원칙 #1).

## TODO 코멘트 정책

- 형식: `TODO(YYYY-MM-DD): <사유 + 후속 작업 위치>`
- 등록 위치: 코드 내 + 작업 단위 commit message
- 회수 정책: 다음 작업 단위에서 우선 처리. 24 시간 이상 잔존 시 사용자에게 상기.
