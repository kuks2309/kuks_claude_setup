# 기술 부채 방지 원칙

기술 부채를 누적하지 않기 위해 지키는 원칙의 단일 근원(SSOT).

## 핵심 원칙: 정공법 우선

- 시간이 더 걸리더라도 **근본 원인 해결**(정공법) 을 선택한다.
- workaround / monkey-patch / 우회 코드 / 임시 hack 은 기본적으로 금지.
- 본 원칙은 [coding.md](coding.md) "지시 회피 대안 제시 금지" 와 충돌 시 **본 원칙이 우선**한다 (우회는 지시 회피의 일종으로 본다). 단, **정공법이 사용자 명시 지시 범위를 초과하는 경우 강제 실행이 아닌 강제 보고** — [coding.md](coding.md) §충돌 해소 우선순위 매트릭스 §정공법이 사용자 지시 범위를 초과할 때 참조.

## 실시간 / 임베디드 시스템에서 정공법이 특히 중요한 이유

- 실시간 시스템에서 workaround(불필요한 sleep, 무한 retry) 는 **타이밍 결정성**을 깨뜨려 추후 디버깅 비용이 폭증한다.
- 메시징 / RPC / pub-sub 계층의 workaround 는 **silent failure** (메시지 drop·재정렬을 못 보는 상태) 로 이어지기 쉽다.
- 의미가 다른 두 상수를 같은 값이라는 이유로 합치는 것은 정공법 위반이다 ([coding.md](coding.md) "상수 분리 원칙").

## 우회 사용 3 조건 (모두 만족 시에만)

정공법이 명백히 비현실적인 경우에 한해 우회를 제안할 수 있으며, 다음을 **모두** 만족해야 한다.

1. 정공법의 비용 / 리스크와 우회 방식을 함께 제시
2. 사용자 명시 승인
3. 우회 사유 + 정리 일정을 코드 주석 또는 이슈 트래커에 기록

## 하드웨어 quirk 우회 (강화 조항)

다음은 정공법이 불가능하거나 가성비가 낮은 경우로, 우회 허용 사례에 명시 포함한다.

- 벤더 펌웨어 / silicon errata 의 알려진 버그
- 외부 SDK / 드라이버 라이브러리 내부 동작 (vendored, read-only)
- 외부 의존성의 알려진 버그 (release notes / issue tracker 에 기재된 것)

이 경우에도 위 3 조건은 모두 적용되며, 우회 코드에는 **벤더 매뉴얼 / 이슈 트래커 링크 + 점검 일정**을 주석으로 남긴다.

## 시간 트레이드오프 보고 의무

- 정공법으로 추가 시간이 발생할 것이 예상되면 **착수 전** 사용자에게 보고한다.
- "빠른 수정" 을 사용자가 명시 요청하지 않은 이상 정공법을 기본으로 한다.
- 정공법이 사용자 명시 지시 범위를 초과하는 구조 변경을 요구할 때는 **자율 실행 금지** — 비용 / 리스크 / 대안을 사용자에게 보고하고 승인 후 진행 (루트 `CLAUDE.md` 핵심 원칙 #2).

## 임시 / 진단 코드 정리

- **기본은 동일 작업 단위에서 제거** — 디버그 print, 임시 로그, 진단 stub.
- 사용자가 명시적으로 "TODO 로 남겨라" / "다음 작업에서 정리" 라고 요청한 경우에 한해 코드 내 `TODO(YYYY-MM-DD): ...` 형식으로 명시하고, 다음 commit 메시지에서 보고한다 (자동 미루기 금지 — 루트 `CLAUDE.md` 핵심 원칙 #1).

## TODO 코멘트 정책

- TODO 만 남기고 작업을 종료하지 않는다 — 최소한 다음 commit 메시지에서 "기존 TODO N 개" 를 보고한다.
- 새 TODO 추가 시 형식: `// TODO(YYYY-MM-DD): <할 일> [참조 파일/이슈]`
- TODO 가 30 일 이상 방치되면 다음 작업 시작 전에 정리하거나 정식 이슈로 승격한다.

## ADR Open Question 관리

- ADR(Architecture Decision Record) 의 OQ(미해결) 항목은 작성 시점부터 **30 일** 내 재평가하여 결정 보완 또는 OQ 닫기 처리한다.
- 30 일 경과 OQ 는 이슈 트래커에 회수 항목으로 등록하거나 해당 ADR 의 상태를 "Superseded" / "Withdrawn" 으로 변경한다.
