# 반복 수정 회피 (Iteration Anti-Pattern)

동일 작업을 부분 수정으로 반복 (iteration loop) 하지 않기 위한 SSOT. [coding.md](coding.md) §작업 범위, [tech_debt.md](tech_debt.md) §정공법 우선 의 **문서·작성·스크립트 영역** 적용판.

## 1 회 정공법 우선

- 사용자 요청에 대해 **첫 작성 전** 아래 4 항목을 점검 후 시작.
- 부분 수정이 2 회 이상 반복되면 추가 수정 대신 §Iteration Loop 탈출 규칙 적용 — 즉시 재작성 금지, 사용자 의도 재확인 우선.

## 1. 기존 자료 우선 조사 (의무)

새 파일 / 새 섹션 작성 전 다음 위치 grep:

- `docs/user_instructions/user_instructions.md` — 동일·유사 요구 이력
- `docs/worklog/` — 과거 작업
- 도메인 폴더 (`code_review/`, `analysis/`, `claude-mistake/` 등)

발견 시 사용자에게 1줄 보고 후 방향 (재사용 / 갱신 / 신규) 확정.

## 2. 사전 자료 + 구조 보고 의무

작성 착수 전 다음 두 가지를 1~3줄로 보고:

- 활용할 기존 자료 (있다면 경로)
- 작성 구조 개요 (헤딩 / 표 / 흐름)

사용자 승인 후 본격 작성.

## 3. 정정 발생 시 임의 추가분 전체 감사

- 사용자가 한 항목을 지적하면 (예: "왜 X 가 들어가나?"), 해당 항목만 제거·수정하지 않는다.
- **사용자가 명시 요청하지 않은 모든 추가분을 동시에 검토** 하여 제거 후보 N 개를 보고.
- 사용자 승인 수신 시에만 제거 — 자동 제거 금지 (CLAUDE.md 핵심 원칙 #1).
- 보고 형식: "지적 항목 제거. 추가 점검 결과 다음 N 개도 임의 추가분 후보로 판단됩니다 — 함께 제거할지 확인 부탁드립니다: ..."

## 4. 단일 파일도 구조 사전 승인

의도가 모호한 단일 파일 작업은 1~3줄 구조 요약 후 승인. 모호 판정 기준:

- 모호 단어: "한 줄로", "간단히", "짧게", "빠르게", "기본적으로"
- 형식 미지정 (헤딩 구조 / 표 / 흐름 등이 사용자 지시에 없음)

본 룰은 [coding.md](coding.md) §사전 승인 없이 진행 가능 (단일 파일 버그 수정 등) 보다 우선 — 모호하면 강화판 적용.

## Iteration Loop 탈출 규칙

2 회째 부분 수정 요청 도착 시:

1. **즉시 추가 수정 중단**.
2. 사용자 의도 재확인 질문 1 회 — 현재 산출물의 어느 부분이 의도와 다른가.
3. 답변 수신 후 **전체 재작성** 으로 전환 (즉시 재작성 금지 — 사용자 지시 재확인 우선).

## 적용 우선순위

- 본 규칙과 [coding.md](coding.md) §작업 범위 의 "지시사항을 회피하려는 대안 제시 금지" 충돌 시 본 규칙 우선 (반복 부분 수정도 회피의 일종).
- [tech_debt.md](tech_debt.md) §정공법 우선 의 **문서·작성·스크립트 영역** 적용판으로 본다.
