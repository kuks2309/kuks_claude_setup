# Iteration 반복 수정 방지 원칙

동일 작업을 여러 번 부분 수정하며 사용자 의도에서 점점 멀어지는 패턴을 방지하기 위한 단일 근원(SSOT).

본 규칙은 [coding.md](coding.md) 및 [workflow.md](workflow.md) 를 강화한다.

## 핵심 원칙: 1 회 정공법 우선

- 사용자 요청에 대해 **첫 작성 전** 아래 4 항목을 점검 후 시작한다.
- 한 번의 정정으로 끝내지 못하고 2 회 이상 부분 수정으로 이어지면, 추가 수정 대신 §Iteration Loop 탈출 규칙 (L38-45) 을 따른다 — **사용자 의도 재확인 질문 1 회 → 답변 수신 후 전체 재작성** (즉시 재작성 금지 — 사용자 지시 재확인 우선).

## 1. 기존 자료 우선 조사 (의무)

- 신규 문서·스크립트·alias·환경 설정 작성 전, 동종 자료가 이미 존재하는지 `grep` / `find` / 워크스페이스 검색을 1 회 이상 수행한다.
- 검색 키워드 예: 파일명, 명령어 단편, 카테고리 키워드, 사용자가 사용한 한국어 표현.
- 동종 자료가 존재하면 **기존 자료를 인용·확장** 하고, 신규 작성하더라도 사용자에게 사전 보고한다.
- 적용 범위: README, 실행 명령어 문서, 설정 파일, alias, 환경 변수, 디버그 스크립트.

## 2. 모호한 단어 추측 금지

- 사용자 메시지에 정량적 기준이 없는 단어 ("한 줄", "간단히", "짧게", "빠르게", "기본적으로") 가 등장하면 **추측하지 않는다**.
- 1 회 질문하여 의도를 확인한 뒤 작성한다.
  - 예: "한 줄 = 단일 명령 라인 / 카테고리 치트시트 중 어느 쪽?"
- 질문 비용 (수 초) 보다 잘못된 작성 후 재작업 비용이 항상 더 크다.

## 3. 정정 발생 시 임의 추가분 전체 감사

- 사용자가 작성물의 한 항목을 지적하면 (예: "왜 X 가 들어가나?"), 해당 항목만 제거·수정하지 않는다.
- **사용자가 명시 요청하지 않은 모든 추가분을 동시에 검토** 하여, 제거 후보 N 개를 보고 후 **사용자 승인 수신 시 제거**. 자동 제거 금지 (루트 `CLAUDE.md` 핵심 원칙 #1 — 사용자가 지시한 것만 수행).
- 보고 형식: "지적 항목 제거. 추가 점검 결과 다음 N 개도 임의 추가분 후보로 판단됩니다 — 함께 제거할지 확인 부탁드립니다: ..."

## 4. 단일 파일도 구조 사전 승인

- 사용자 의도가 모호한 경우 단일 마크다운·스크립트 파일이라도 **1~3 줄 구조 요약** 후 승인을 받는다.
- 구조 요약 항목: 파일 형식 / 포함될 주요 항목 / 제외할 항목.
- 의도가 명확한 경우 (사용자가 형식까지 지정한 경우) 는 본 단계 생략 가능.

## Iteration Loop 탈출 규칙

작성 → 사용자 지적 → 재작성 사이클이 **2 회 반복** 되면, 다음 사이클 전에 다음 단계를 거친다.

1. 작성 중지
2. 사용자 원 메시지 재독 — 핵심 단어 1~3 개 식별
3. 단어 정의를 사용자에게 1 회 질문 (이미 질문했더라도 재확인)
4. 답변 수신 후 **전체 재작성** (부분 수정 금지)
