# 코드 작업 규칙

코드 작성·수정 시 지켜야 할 규칙의 단일 근원.

## 금지 사항

- 사용자가 요청하지 않은 모델/아키텍처 구조 변경 금지
- 요청 범위를 벗어난 코딩 금지
- 지시사항을 회피하려는 대안 제시 금지 (단, 지시 먼저 수행한 후 1줄 대안 제안은 허용). 본 금지의 영역별 override 는 §충돌 해소 우선순위 매트릭스 참조 — 정공법 우회 분기는 [tech_debt.md](tech_debt.md), 반복 수정 분기는 [iteration_anti_pattern.md](iteration_anti_pattern.md) 가 우선.

## 코드 수정 규칙

- 기존에 지시·승인된 코드 로직을 임의로 변경하지 않는다.
- 변경이 필요하면 반드시 사유를 설명하고 승인받는다.
- 수정 전 현재 코드 구조를 먼저 파악한다.
- 중복 코드, 중복 함수, 중복 변수가 있는지 조사하고 보고한다.

## 상수 분리 원칙

의미가 다른 두 물리량/도메인 값이 우연히 같은 값으로 보일 때 하나의 상수로 합치지 않는다.

- 우연한 동치는 펌웨어 / 하드웨어 / 단위 변환에서 흔하다 (예: 인코더 펄스 수와 디지털 풀스케일이 비슷한 자릿수).
- 합치면 한쪽 값이 변하는 순간 silent bug 가 발생하며, 디버깅 비용이 매우 크다.
- 별도 이름·별도 상수로 정의하고, 각 사용처에서 의미를 주석으로 명시한다.
- 외부 매뉴얼·데이터시트의 모호한 수치를 추정으로 단정 적용하지 않는다 — 실측·실험으로 검증한다 ([manual.md](manual.md) "추정 금지 · 실측 검증").

## 사전 승인이 필요한 변경 (트리거)

다음 변경은 코딩 전에 구조를 제시하고 사용자 승인을 받는다.

- 새 패키지 / 모듈 신규 생성 또는 삭제
- 외부에 노출되는 인터페이스(API (Application Programming Interface), 메시지/서비스 타입, 토픽 이름, 함수 시그니처) 변경
- 빌드 시스템 / 의존성 / 설치 규칙 변경
- 하드웨어 인터페이스 (통신 속도, 핀맵, 펌웨어 통신 파라미터) 변경
- 데이터 스키마 / 마이그레이션 / 영속성 포맷 변경

## 사전 승인 없이 진행 가능한 변경

- 단일 파일 내 버그 수정 / 로그 개선 / 주석 보강
- 파라미터 추가 (기본값 보존, 외부 인터페이스 미변경)
- 내부 함수 리팩터 (외부 시그니처 미변경)
- **단, 사용자 의도가 모호한 경우 [iteration_anti_pattern.md](iteration_anti_pattern.md) §4 가 우선** — 1~3 줄 구조 요약 후 승인. 모호 판정 기준은 [iteration_anti_pattern.md](iteration_anti_pattern.md) §2 (모호 단어 "한 줄"/"간단히"/"짧게"/"빠르게"/"기본적으로").

## 코딩 스타일 (formatter / 컨벤션)

- **프로젝트가 채택한 formatter 컨벤션을 따른다.** `.clang-format`, `.editorconfig`, `pyproject.toml` 등 저장소 내 설정 파일이 있으면 그것이 본 문서보다 우선한다.
- 새 워크스페이스에서 컨벤션이 정해지지 않았다면, 워크스페이스 루트 `CLAUDE.md` 또는 모듈 CLAUDE.md 에 한 번 명시하고 그 이후 일관 적용한다 (예: C/C++ Allman vs K&R, Python `black` 등).
- 본 규칙은 새로 작성하는 코드부터 강제. 기존 코드는 수정 시 함께 reformat.
- 모듈 CLAUDE.md 또는 `docs/` 도메인 문서에 명시된 컨벤션이 본 문서보다 우선.

## 네이밍·기타 컨벤션

- 들여쓰기·네이밍·언어별 컨벤션은 프로젝트별 / 모듈별 규약을 따른다.

## 보고 양식 (변경 분기에서만)

다음 시점에만 다짐과 확인을 보고한다 (매 답변 강제 X).

1. 사전 승인 트리거에 해당하는 변경을 시작할 때
2. 기존 로직을 변경할 때 (사유 + 영향 범위)
3. 우회 / workaround 를 사용할 때 ([tech_debt.md](tech_debt.md) 3 조건)
4. 정공법 진행으로 추가 시간이 예상되는 경우 ([tech_debt.md](tech_debt.md) §시간 트레이드오프 보고 의무)

일상적인 단순 작업(파라미터 추가, 로그 보강 등)에서는 결과만 간결히 보고한다.

## 충돌 해소 우선순위 매트릭스

본 폴더의 모든 ★ 규칙 (`coding.md`, `tech_debt.md`, `iteration_anti_pattern.md`) 은 루트 `CLAUDE.md` 핵심 원칙 #1 "사용자가 지시한 사항만 수행" / #2 "임의로 기능을 추가하거나 변경하지 않는다" 의 **하위 적용 규칙**이다. 어떤 ★ 규칙도 루트 핵심 원칙을 override 하지 않는다.

`coding.md` 가 base. `tech_debt.md` 와 `iteration_anti_pattern.md` 는 base 위의 영역별 강화판. 셋이 충돌하면 다음 표에 따른다.

| 작업 영역 | 1 차 SSOT (우선) | 2 차 SSOT | 비고 |
|---|---|---|---|
| 사용자 명시 지시 범위 안 일반 코드 작업 | `coding.md` | — | 다른 두 SSOT 는 추가 제약만 부과 |
| 우회 / workaround / monkey-patch 사용 분기 | [tech_debt.md](tech_debt.md) §우회 3 조건 | `coding.md:9` | "회피금지 1 줄 대안 제안 허용" 보다 `tech_debt.md` 가 강함 |
| 동일 작업 2 회째 사용자 지적 후 재작성 분기 | [iteration_anti_pattern.md](iteration_anti_pattern.md) §탈출 규칙 | `coding.md:9` | 반복 부분 수정도 회피의 일종 |
| 단일 파일 작업 — 사용자 의도 **명확** | `coding.md` §사전 승인 없이 진행 가능 | — | 코드 / 문서 / 스크립트 동일 |
| 단일 파일 작업 — 사용자 의도 **모호** | [iteration_anti_pattern.md](iteration_anti_pattern.md) §4 | `coding.md` §사전 승인 없이 진행 가능 | 의도 모호 시 강화판 우선 |
| 정공법이 사용자 지시 범위를 **초과** | 사용자 보고 + 승인 (강제 보고) | [tech_debt.md](tech_debt.md) §정공법 우선 | 범위 초과 = 핵심 원칙 #2 와 충돌, 강제 실행 X |
| 정공법으로 추가 시간 예상 | [tech_debt.md](tech_debt.md) §시간 트레이드오프 보고 의무 | `coding.md` 보고 양식 4 번 | |
| 임시 / 진단 코드 정리 시점 | [tech_debt.md](tech_debt.md) §임시/진단 코드 default = 동일 작업 단위 | 사용자 명시 "TODO 로 남겨라" 시에만 미루기 허용 | 자동 미루기 금지 (핵심 원칙 #1) |
| 상수 분리 / 동치 의미 분리 | `coding.md` §상수 분리 + [tech_debt.md](tech_debt.md) | — | 양 문서 일관 |
| 외부 자료 / 매뉴얼 인용 정확도 | [manual.md](manual.md) | `coding.md:25` | `manual.md` 가 datasheet 인용 SSOT |

### 단일 파일 작업 case-by-case 분기

| 파일 종류 | 의도 명확 | 의도 모호 |
|---|---|---|
| 코드 파일 (`.c`, `.cpp`, `.py`, `.cc`, `.h`) | 사전 승인 없이 진행 | 1~3 줄 구조 요약 후 승인 |
| 문서 파일 (`.md`, `.rst`, `.txt`) | 사전 승인 없이 진행 | 1~3 줄 구조 요약 후 승인 |
| 스크립트 파일 (`.sh`, alias, 환경 설정) | 사전 승인 없이 진행 + 기존 자료 grep 1 회 의무 | 1~3 줄 구조 요약 후 승인 |
| 외부 인터페이스 (`.proto`, `.msg`, `.srv`, OpenAPI) | 명확해도 사전 승인 필요 | 사전 승인 필요 |
| 빌드 / 의존성 (`CMakeLists.txt`, `package.xml`, `pyproject.toml`) | 명확해도 사전 승인 필요 | 사전 승인 필요 |

분기 알고리즘: (1) 파일이 외부 인터페이스/빌드/의존성/HW/스키마면 무조건 사전 승인. (2) 그 외 단일 파일 작업에서 모호 단어가 있거나 형식 미지정이면 1~3 줄 구조 요약. (3) 의도 명확 (형식·결과 지정 또는 단일 식별자 변경) 시 진행 후 결과 보고.

### 정공법이 사용자 지시 범위를 초과할 때

사용자가 "이 한 줄만 고쳐줘" 라고 했는데 근본 원인이 다른 모듈에 있는 구조 문제인 경우, `tech_debt.md` 정공법을 **자율적으로 실행하지 않는다** — 루트 핵심 원칙 #2 위반.

1. 사용자 지시 범위 내 표면 수정의 효과·리스크를 1 줄로 보고.
2. 정공법 비용 / 리스크 / 대안을 2~3 줄로 함께 보고 (1 줄 대안 한도 — `coding.md:9` 허용 범위 안).
3. 사용자 선택 수신.
4. 사용자가 정공법 승인 시에만 범위 확장. 미승인이면 표면 수정만.

본 절차가 `tech_debt.md` §정공법 우선의 **강제 실행** 해석을 **강제 보고** 해석으로 정정한다.
